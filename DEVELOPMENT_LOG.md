# DEVELOPMENT_LOG.md

# Time Entry Portfolio Lab 開発ログ

## 目的

本レポジトリは、時間固定エントリー・時間固定決済を中心としたFXポートフォリオ戦略を管理・検証するための開発ログである。

現在の主な目的は以下。

- 現行運用中の時間手法ポートフォリオをPythonで再現する
- 各ロジックの条件、除外日、SL/TP、スプレッド設定を明文化する
- バックテスト結果と実運用結果の差異を確認する
- 追加フィルタの検証により、PF・DD・RoMDの改善可能性を調査する
- 過剰最適化を避けるため、IS/OOS・年別・月別・ロジック別に安定性を確認する

---

## 現在の対象ポートフォリオ

### マスター16ロジック

対象通貨ペア：

- USDJPY
- EURJPY
- GBPJPY
- AUDJPY

主な特徴：

- JST基準の固定時間エントリー
- JST基準の固定時間決済

- 年末年始停止
- 重要指標日停止
- 月別・日付別の個別停止条件あり

### 追加オージー絡みロジック

対象通貨ペア：

- AUDUSD

- GBPAUD

主な特徴：

- AUDUSDは毎月9日〜15日、および25日〜月末も稼働
- RBA、豪CPI、FOMC、ECB、BOEなどのイベント停止を使用
- FOMCは前日・当日停止が必要

対象通貨ペア：

- EURAUD
- GBPAUD

主な特徴：

- JST固定時間エントリー

- 指標停止と季節性停止を使用

## 2026-06-13 メモ

### 確認済み修正点

#### UJ_Short_Core

既存Pythonコードでは、通常日のエントリー時刻が `07:53` になっていたが、正しくは `08:04`。

修正前：

```python
else: en_h, en_m = 7, 53
```

修正後：

```python
else: en_h, en_m = 8, 4
```

また、UJ_Short_Coreの通常日は以下が正しい。

```text
通常日：SL50 / TPなし
ゴトー日：SL20 / TP50
```

そのため、UJ_Short_Coreでは日付条件によりSL/TPを分岐させる必要がある。

## 検証エンジンに関する注意点

### 1. スプレッド処理

現行コードでは、エントリー価格を不利にすることでスプレッドを反映している。

```python
ep = o[s_idx] + spread if is_long else o[s_idx] - spread
```

この方式は、トレードごとにスプレッド分を1回控除する考え方に近い。

今後の改善候補：

- `entry_adjust`方式として明記する
- Bid/Ask方式に変更したバージョンと比較する
- 通貨ペアごとのスプレッド設定を外部CSVで管理する

---

### 2. SL/TP同一足内判定

現行コードでは、1分足の同一足内でSLとTPの両方に到達した場合、SLを優先する。

これは保守的な判定であり、検証上は安全寄り。

今後の方針：

```text
same_bar_policy = sl_first
```

として明記する。

---

### 3. 指標フィルター

現行コードでは、基本的にエントリー日の日付で指標停止を判定している。

ただし、仕様上は以下が望ましい。

```text
Entry日時〜Exit日時の保有期間に重要指標日が含まれる場合は見送り
```

特に翌日持ち越しロジックでは、エントリー翌日の指標を見逃す可能性がある。

今後の改善候補：

- エントリー日だけでなく、保有期間内の日付をチェックする
- 可能なら指標発表時刻も管理する
- 最低限、保有期間内にイベント日が含まれるかを判定する

---

### 4. FOMC前日停止

オージー絡みロジックでは、FOMC当日だけでなく前日も停止する仕様。

今後の実装方針：

```text
FOMC_STOP_DATES = FOMC当日 + FOMC前日
```

としてイベントフィルターに追加する。

---

### 5. 2026年カレンダー

2026年の経済指標カレンダーは別途保有あり。

今後の方針：

- 2026年イベント日を追加
- 可能ならイベント日をPythonコード直書きではなくCSV管理へ移行する
- 年次更新しやすい形にする

## 運用上の注意

現行手法はすでに実運用中のため、改良版をすぐ本番置き換えしない。

方針：

```text
現行版 = baseline
修正版 = engine_fix
フィルタ版 = experimental
```

として分けて管理する。

フィルタ版は、以下を確認してから採用候補とする。

- ISで改善しているか
- OOSで崩れていないか
- 年別で極端な偏りがないか
- 月別で一部期間だけに依存していないか
- トレード数が減りすぎていないか
- 合計pipsだけでなくMax DDとRoMDが改善しているか
- 実運用期間のDDを後付けで避けるだけのフィルタになっていないか
